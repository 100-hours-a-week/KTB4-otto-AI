## 1. 한 일 — RAG 3대 요구사항

### 1) RAG 파이프라인 → LangChain 마이그레이션
6주차에 직접 구현했던 RAG(`rag_chroma`: ChromaDB + SentenceTransformer + 커스텀 generator)를
**LangChain LCEL**로 옮겼다. (`03_rag/langchain_rag/app/rag_chain.py`)

```
질문 ─▶ retriever ─▶ format_docs ─▶ ChatPromptTemplate ─▶ LLM ─▶ StrOutputParser ─▶ 답변
```

- 검색·프롬프트·모델·파서를 전부 `Runnable`로 연결 (`retriever | prompt | llm | parser`)
- `RunnablePassthrough.assign`으로 검색을 1회만 하고 답변+출처에 공유
- 마이그레이션 전 원본은 `03_rag/legacy/`에 보존 → before/after 비교 가능
- 도중에 LangChain이 **0.3 → 1.x 메이저 업데이트**라 의존성을 올렸는데, 코드 수정 없이 그대로 동작했다.

### 2) FastAPI REST API
`app/api.py`로 6개 엔드포인트 제공: `POST /query`, `POST /chat`(멀티턴), `GET /search`,
`POST /reindex`, `GET /health`, `POST /chat/reset`. 서버 띄우고 `curl`로 실동작 확인.

### 3) LangSmith Tracing + Dataset 평가
- `config.py`에서 `LANGCHAIN_TRACING_V2`를 켜면 모든 체인 실행이 자동 기록
- `evaluation/eval_langsmith.py`: LangSmith Dataset 생성·업로드 + `evaluate`(correctness, source_match)
- 평가셋 11문항(`dataset.jsonl`, "문서에 없음" 케이스 포함)
- **보너스**: RAGAS 4지표(Faithfulness/Answer Relevancy/Context Precision·Recall) 하네스도 추가

## 2. 지난주(6주차)보다 발전한 점

| | 6주차 | 7주차 |
|---|---|---|
| RAG 구현 | 직접 구현(ChromaDB+ST) | **LangChain LCEL 표준화** |
| 대화 | 단발성 Q&A | **멀티턴**(RunnableWithMessageHistory + 기록 트리밍) |
| 평가 | RAGAS만 | LangSmith Dataset + RAGAS |
| 나만의 모델 | 57M (위키만) | **318M**(데이터 확장 + 직접 스케일업) |
| 시스템 | RAG 챗봇 | **도구 쓰는 에이전트**(라우팅+계산+검색+RAG) |
| 안정성 | — | **환각 차단 안전망**, 오프라인 폴백 |

특히 모델을 **57M → 318M(GPT-2 medium급)** 으로 직접 키워서, 사전학습 perplexity가
**31 → 4.4**로 크게 좋아졌다. "내 모델"이 한 단계 진짜로 똑똑해진 게 가장 큰 성취.

## 3. 진행하며 가졌던 의문점

- **"학습을 오래 돌리면 똑똑해지는 거 아닌가?"**
  → 아니었다. 위키로 40k step 학습 후 9,500 step을 더 돌렸는데 val_loss가 **3.42에서 1도 안 움직였다.**
  수렴하면 시간을 더 부어도 0이라는 걸 직접 봤다. **시간이 아니라 파라미터(용량)가 천장**이었다.

- **"그럼 모델을 키우면 다 좋아지나?"**
  → 부분적으로만. 318M으로 키우니 사전학습 loss(유창성·지식)는 확 좋아졌는데,
  **도구 라우팅 같은 단순 task에선 57M과 별 차이가 없었다.** 베이스의 향상이 모든 task에
  똑같이 나타나지 않는다는 걸 배웠다.

- **"작은 모델이 헛소리(환각)하는 건 어떻게 막지?"**
  → 모델 자체는 못 고친다. 대신 **모델의 자유 생성 텍스트를 답으로 안 쓰고**, 도구 호출 결정만
  믿고 실패하면 규칙으로 폴백하게 했다. "모델을 고치는 게 아니라 시스템으로 감싼다"는 관점.

- **무료 API quota 제약 속 개발** — Gemini 무료티어 일일 한도(20회)에 계속 막혔다.
  로컬 문서 검색(TF-IDF)·규칙 라우터로 **외부 의존성을 떼어내** quota 없이도 돌게 만든 게 교훈.

- **숫자 복사가 자꾸 틀린다**(`345+678`→`346+678`). 모델을 키워도 안 고쳐졌다.
  → 토크나이징 문제(용량 무관). 에이전트가 **원문에서 숫자를 다시 추출**해 보완.

## 4. 앞으로 발전시키고 싶은 방향

- **observation→answer 합성**: 도구가 가져온 사실을 318M의 유창성으로 자연스럽게 다듬기
  (근거 기반이라 환각 불가능 — 318M을 안전하게 쓰는 자리)
- **quota 없는 진짜 벡터 DB**: ChromaDB를 로컬 임베딩(sentence-transformers)으로 돌려
  Gemini 의존 제거
- **Advanced Retrieval**: Reranking(Cross-Encoder), Multi-Query, Hybrid Search 적용
- **측정 기반 개선 루프**: RAGAS baseline을 잡고 → 바꾸고 → 재측정으로 "감"이 아닌 수치로 개선
- **더 큰 모델 / 큰 오픈모델 파인튜닝**: from-scratch의 한계를 넘으려면 결국 더 큰 용량 필요
- **멀티스텝 에이전트**: 도구 결과를 보고 다음 행동을 스스로 결정하는 루프
