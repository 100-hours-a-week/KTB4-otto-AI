# 2주차 과제 회고

 처음에는 HTTP, REST API, FastAPI, 데이터베이스, LLM 연동이라는 단어들이 각각 따로 떨어져 있는 개념처럼 느껴졌습니다. 저는 아직 코딩 경험이 많지 않은 비전공자에 가깝기 때문에, 과제를 시작할 때는 무엇부터 만들어야 하는지, 어떤 파일을 수정해야 하는지, 터미널에서 어떤 명령어를 입력해야 하는지도 쉽게 떠오르지 않았습니다.

처음 과제를 시작했을 때는 AI의 기여도가 매우 높았습니다. API 구조를 어떻게 잡아야 하는지, FastAPI 파일을 어떻게 나누어야 하는지, GitHub에 어떻게 올려야 하는지 대부분 AI의 안내를 받으며 진행했습니다. 하지만 단순히 답을 복사해서 끝내기보다는, 왜 이 명령어를 쓰는지, 왜 이 파일을 나누는지, 왜 이 기능은 백엔드이고 프론트엔드는 아닌지 계속 질문하면서 이해하려고 했습니다.

반복적인 질문과 실습을 거치면서 한 번 물어보고 알게 된 답은 다시 비슷한 상황이 왔을 때 조금씩 제 머릿속에서도 떠오르기 시작했습니다. 처음에는 `uvicorn main:app --reload`가 무엇을 하는지도 낯설었지만, 이제는 FastAPI 서버를 실행하는 명령어라는 것을 이해하게 되었습니다. `/docs`에서 API를 테스트하는 과정, `git add → git commit → git push`로 GitHub에 올리는 과정, 서버를 껐다 켜도 데이터가 남아 있는지 확인하는 과정도 처음보다 훨씬 익숙해졌습니다.

이번 과제에서는 글 CRUD와 댓글 CRUD를 구현하면서 HTTP Method와 URL 설계의 관계를 조금씩 이해할 수 있었습니다. 처음에는 `GET`, `POST`, `PUT`, `PATCH`, `DELETE`가 단순히 외워야 하는 단어처럼 느껴졌지만, 직접 API를 만들면서 `GET /posts`는 글 목록 조회, `POST /posts`는 글 작성처럼 URL은 자원을 표현하고 Method는 행동을 표현한다는 점을 체감했습니다. 이 부분은 REST API를 이해하는 데 가장 중요한 경험이었습니다.

처음에는 Python 리스트를 사용해 데이터를 메모리에 저장했습니다. 이 방식은 구조를 이해하기에는 쉬웠지만, 서버를 재시작하면 데이터가 사라지는 한계가 있었습니다. 이후 sqlite3를 적용하면서 글과 댓글 데이터를 SQLite 데이터베이스 파일에 저장하도록 수정했습니다. `SELECT`, `INSERT`, `UPDATE`, `DELETE` 같은 SQL 문이 실제 API 요청과 연결되는 과정을 보면서, 데이터베이스가 왜 필요한지 조금 더 현실적으로 이해할 수 있었습니다. 특히 서버를 껐다가 다시 켠 뒤에도 제가 작성한 테스트 글이 남아 있는 것을 확인했을 때, 메모리 저장과 데이터베이스 저장의 차이가 분명하게 느껴졌습니다.

추가로 Ollama gemma4를 사용해 로컬 LLM 요약 API도 구현했습니다. 처음에는 LLM을 단순히 웹 화면에서 대화하는 도구로만 생각했지만, 이번 과제를 통해 백엔드 서버가 로컬 LLM에 HTTP 요청을 보내고, 그 결과를 다시 JSON 응답으로 반환할 수 있다는 것을 알게 되었습니다. `POST /posts/{post_id}/summary` API를 만들면서 백엔드가 데이터베이스와 LLM 사이를 연결하는 중간 역할도 할 수 있다는 점을 배웠습니다.

구조 개선 과정도 의미 있었습니다. 처음에는 모든 코드를 `main.py`에 넣어도 된다고 생각했지만, 기능이 늘어나면서 파일을 나누는 이유를 이해하게 되었습니다. Router는 API 경로를 담당하고, Controller는 요청 처리 흐름을 담당하고, Model은 데이터 관리를 담당하도록 나누면서 코드의 책임이 조금 더 명확해졌습니다. 이번 리팩토링은 기능을 바꾸기 위한 작업이 아니라, 기능은 유지하면서 코드 구조를 더 이해하기 쉽게 만드는 작업이라는 점을 배웠습니다.

이번 과제를 하면서 AI는 저에게 단순히 정답을 대신 작성해주는 도구라기보다는, 제가 모르는 부분을 질문하고 확인하면서 학습 흐름을 잡아주는 보조 역할에 가까웠습니다. 처음에는 AI의 안내를 거의 그대로 따라가는 비중이 컸지만, 반복적인 질문과 실습을 통해 점점 “이 상황에서는 이 명령어를 써야겠다”, “이 오류는 서버가 꺼져 있어서 생긴 것 같다”, “이 기능은 프론트엔드가 아니라 백엔드 API의 역할이다”처럼 제 머릿속에서도 연결되는 부분이 생겼습니다. 아직 완전히 혼자 구현할 수 있는 수준은 아니지만, 적어도 같은 문제를 다시 마주했을 때 이전보다 덜 막막하게 접근할 수 있게 되었다는 점에서 의미 있는 과정이었습니다.

또한 이번 주에는 단순히 과제를 제출하는 것뿐만 아니라, 제가 실제로 얼마나 꾸준히 공부하고 있는지도 확인해보고 싶었습니다. 그래서 ZEP에 접속해 있는 시간을 자동으로 계산하는 프로그램도 따로 만들어보았습니다. 이 프로그램은 하루 동안 ZEP 학습 공간에 머문 시간을 기록해서, 제가 실제로 공부에 투자한 물리적인 시간을 확인할 수 있도록 도와줍니다.

처음에는 공부를 많이 했다고 느껴도 실제로 얼마나 집중해서 앉아 있었는지 정확히 알기 어려웠습니다. 그래서 학습 시간을 직접 손으로 기록하는 대신, 접속 시간을 자동으로 계산해 하루하루 공부 시간을 확인하는 방식으로 바꾸고 싶었습니다. 이를 통해 막연히 “오늘 공부했다”라고 생각하는 것이 아니라, 실제 숫자로 제 학습 시간을 확인하고 조금씩 늘려가려는 목표를 세울 수 있었습니다.

ZEP 접속 시간 기록 프로그램은 아직 완성도 높은 프로그램은 아니지만, 제가 필요하다고 느낀 문제를 코드로 해결해보려 했다는 점에서 의미 있는 시도였습니다. 백엔드 과제를 하면서 배운 서버, 요청, 응답, 데이터 저장 개념뿐만 아니라, 실제 제 학습 습관을 개선하기 위한 작은 도구를 직접 만들어보며 개발을 더 현실적인 문제 해결 방식으로 바라보게 되었습니다.

물론 아직 부족한 점도 많습니다. SQL 문법은 아직 익숙하지 않고, 예외 처리도 더 다양한 상황을 고려해야 한다고 느꼈습니다. LLM 연동도 현재는 단순한 요약 기능 수준이기 때문에, 프롬프트를 어떻게 더 안정적으로 작성할지, 응답이 이상할 때 어떻게 처리할지 더 배워야 합니다. 또한 프론트엔드와 백엔드를 연결하는 흐름은 아직 직접 구현하지 않았기 때문에, 나중에는 HTML/CSS/JS 또는 Streamlit을 사용해서 사용자가 보는 화면과 백엔드 API를 연결하는 end-to-end 흐름도 경험해보고 싶습니다.

앞으로는 코드를 보기만 하는 것이 아니라 직접 더 많이 수정해보면서 구조를 익히고 싶습니다. Router, Controller, Model이 각각 어떤 책임을 가지는지 더 명확히 이해하고, 기능이 늘어났을 때 어떤 기준으로 파일을 나눌지 연습하고 싶습니다. 또한 데이터베이스, 예외 처리, API 설계, Git 사용법도 반복해서 익숙해져야 한다고 느꼈습니다.

이번 1주일은 완벽한 코드를 만든 시간이라기보다, 백엔드가 어떻게 요청을 받고, 데이터를 저장하고, LLM을 호출하고, 다시 응답을 반환하는지 전체 흐름을 처음으로 경험한 시간이었습니다. 아직 AI의 도움을 많이 받는 단계이지만, 반복해서 질문하고 직접 실행해보면서 조금씩 제 머릿속에도 흐름이 남기 시작했습니다. 그래서 이번 과제는 저에게 단순한 제출물이 아니라, 앞으로 개발을 계속 배워나가기 위한 중요한 출발점이 되었습니다.
